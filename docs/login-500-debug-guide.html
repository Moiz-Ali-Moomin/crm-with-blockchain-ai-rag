🔴 Active Investigation — Issue #058

Login Endpoint Returns 500
Instead of 401

POST /api/v1/auth/login returns 500 "Database error occurred" for every request. The auth service itself is correct. The bug lives somewhere in the NestJS HTTP pipeline. This guide documents the full investigation and the exact commands to resolve it.

NestJS v10 Prisma AsyncLocalStorage AllExceptionsFilter Docker — Hetzner April 2026
1
Symptom & Impact
🚨
Production — Login Completely Broken
Every login attempt returns HTTP 500. No user can log in. The frontend shows a generic network error. The expected response is 401 with "Invalid email or password".
Observed HTTP response
{
  "success": false,
  "statusCode": 500,
  "message": "Database error occurred",   // ← the key clue
  "timestamp": "2026-04-07T...",
  "path": "/api/v1/auth/login"
}

Expected:
{
  "statusCode": 401,
  "message": "Invalid email or password"
}
2
What "Database error occurred" Actually Means
The message is not generic — it comes from exactly one code path in the global exception filter. Understanding this immediately constrains the search space.
src/common/filters/all-exceptions.filter.ts — mapPrismaError() 
private mapPrismaError(exception: Prisma.PrismaClientKnownRequestError) {
  switch (exception.code) {
    case 'P2002': return { status: 409, message: 'A record with this field already exists' };
    case 'P2025': return { status: 404, message: 'Record not found' };
    case 'P2003': return { status: 400, message: 'Foreign key constraint failed' };
    case 'P2014': return { status: 400, message: 'Cannot delete — has related records' };
    default:
      return { status: 500, message: 'Database error occurred' }; // ← WE ARE HERE
  }
}
What this tells us:
  • The exception IS a Prisma.PrismaClientKnownRequestError — not a plain Error, not an HttpException, not a DomainError
  • Its .code is not P2002, P2025, P2003, or P2014 — some other Prisma error code
  • In production, unhandled non-Prisma errors return "An unexpected error occurred" — so this cannot be that path
  • The DomainError check runs before the Prisma check — so UnauthorizedError is NOT being thrown (or its instanceof check is failing)
💡
Key exception filter order
HttpException → DomainError → ZodError → PrismaClientKnownRequestError → PrismaClientValidationError → default
Reaching the Prisma branch means all three checks above it were false.

HTTP Pipeline Execution Order for Login
Every NestJS HTTP request for POST /api/v1/auth/login passes through these layers in order. The Prisma error must be thrown by one of them.
1
Express middleware: RequestIdMiddleware — Injects X-Request-ID header
✅ Cleared
2
Express middleware: TenantContextMiddleware — Excluded for api/v1/auth/(.*) routes via .exclude()
✅ Cleared
3
Global Guard: JwtAuthGuard — Route is @Public(), returns true immediately — Passport strategy never invoked
✅ Cleared
4
Global Guard: ThrottlerGuard — Uses in-memory storage (ThrottlerModule.forRoot has no Redis storage configured), no Prisma calls
✅ Cleared
5
Global Interceptor: ResponseTransformInterceptor — Wraps successful responses in { success, data, timestamp }. Error path is unaffected.
✅ Cleared
6
Global Pipe: ZodValidationPipe (no schema) — Passthrough. Param-level pipe validates LoginSchema; bad input → BadRequestException (400), not Prisma error.
✅ Cleared
7
Controller handler: AuthController.login() → AuthService.login()
⚠ Suspected
8
Auth service layer — authRepo.findByEmail() → withoutTenantScope() → prisma.user.findFirst()
⚠ Suspected
9
Global Exception Filter: AllExceptionsFilter — Catches the thrown error, maps it, logs stack trace
→ Returns 500

Full Pipeline Audit Results
  • ✅
    RequestIdMiddleware — Only sets x-request-id header. No database access.
    Cleared
  • ✅
    TenantContextMiddleware — Excluded for api/v1/auth/(.*). Even if it ran: sets tenantId: '', which withoutTenantScope() overrides with null.
    Cleared
  • ✅
    JwtAuthGuard — Checks @Public() via Reflector, returns true immediately. super.canActivate() is never reached. Passport's JwtStrategy.validate() is never called.
    Cleared
  • ✅
    ThrottlerGuard — Default in-memory storage. No Redis, no Prisma. Throws ThrottlerException (HttpException → 429) if rate-limited, which would be caught before Prisma check.
    Cleared
  • ✅
    ResponseTransformInterceptor — Only wraps successful response data. Errors are not intercepted here (NestJS exception filters handle errors, not interceptors).
    Cleared
  • ✅
    AuditLogInterceptor — NOT global. Only applied via @UseInterceptors(AuditLogInterceptor) on specific routes. Login route has no such decorator. Even if it ran, it skips when !user?.tenantId.
    Cleared
  • ✅
    ZodValidationPipe (global) — No schema → passthrough. Parameter-level pipe for @Body throws BadRequestException on invalid input → would produce 400, not 500.
    Cleared
  • ✅
    AuthService.login() — Directly tested via docker exec crm_api node -e "..." with ALS context set. Throws UnauthorizedError correctly for wrong credentials. Service itself is not the bug.
    Cleared
  • ✅
    AuthRepository.findByEmail() — Uses withoutTenantScope() which sets tenantId: null in ALS. Extension checks !null === true → bypasses scoping → calls raw query(args).
    Cleared
  • ✅
    Prisma extension $allOperations — Bypasses tenant injection for: empty/null/undefined tenantId, and models in UNSCOPED_MODELS (Tenant, BillingInfo). withoutTenantScope guarantees null tenantId.
    Cleared
  • ✅
    DomainError instanceof check — tsconfig.json target is ES2021, so new.target is native. Object.setPrototypeOf(this, new.target.prototype) in DomainError correctly fixes the prototype chain. instanceof works.
    Cleared
  • ✅
    AllExceptionsFilter ordering — DomainError check is at line 88, Prisma check is at line 108. DomainError runs first. If UnauthorizedError were thrown, it would produce 401, not 500.
    Cleared
  • ⚠️
    Actual Prisma error code — Unknown. The code is not P2002/P2025/P2003/P2014. Could be P1001 (connection), P2016 (query parse), schema mismatch, or something else entirely. Must be read from the stack trace.
    Unknown
  • ⚠️
    DB schema vs Prisma client mismatch — If a migration failed silently or the Prisma client was generated against a different schema version, queries could fail before generating SQL. This would explain why no SQL reaches Postgres.
    Not ruled out
5
Get the Stack Trace — End This in 5 Minutes
✅
AllExceptionsFilter Already Logs Everything
Every 500 error logs exception.message and exception.stack to the container's stdout. The stack trace will show the exact file, line, and Prisma code. No code changes needed — just read the logs.
Step 1 — SSH into the server
bash
ssh root@37.27.41.226
Step 2 — Trigger a login request
Any credentials will trigger the error — even wrong ones will reproduce the 500 if the bug is pre-auth-check. Even correct ones will if the bug is post-auth.
bash
curl -s -X POST http://localhost:3001/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"test@test.com","password":"wrongpassword"}' | jq .
Step 3 — Read the stack trace from container logs
bash
# Most recent 150 lines — the error log will be near the end
docker logs crm_api --tail=150 2>&1

# If logs are JSON, grep for the stack field
docker logs crm_api --tail=150 2>&1 | grep -A 40 '"stack"'

# Or search for the auth path specifically
docker logs crm_api --tail=150 2>&1 | grep -A 40 'auth/login'
Step 4 — Direct Prisma isolation test (no HTTP pipeline)
Tests whether the Prisma error is in findFirst itself, independent of NestJS.
bash
docker exec crm_api node -e "
const { PrismaClient } = require('@prisma/client');
const p = new PrismaClient();
p.user.findFirst({ where: { email: 'test@test.com' } })
  .then(r => {
    console.log('Result:', r ? 'user found' : 'null (not found)');
    p.\$disconnect();
  })
  .catch(e => {
    console.log('ERROR type:', e.constructor.name);
    console.log('ERROR code:', e.code);
    console.log('ERROR msg:', e.message);
    p.\$disconnect();
  });
"
Step 5 — Check for Prisma/DB schema mismatch
If the Prisma client schema doesn't match the live DB, queries fail before SQL is generated.
bash
# Check migration status inside running container
docker exec crm_api npx prisma migrate status

# Or check directly in the DB
docker exec crm_postgres psql -U crm_user -d crm_db \
  -c "SELECT migration_name, finished_at FROM _prisma_migrations ORDER BY started_at DESC LIMIT 10;"
⚠️
What to look for in the stack trace
The stack trace will show: (1) the Prisma error code (e.g. P1001, P2016, P2034…), (2) which file threw it (prisma.service.js line X, auth.repository.js line Y), and (3) at what point in the auth flow (before or after the password check). That's all that's needed to identify and fix the bug.

Related Fixed Issues (Same Session)
Issue #057 — crm_api boot crash: OPENAI_API_KEY Fixed
CopilotService used configService.getOrThrow('OPENAI_API_KEY') which crashes the whole app if the key is absent. Fixed by changing to configService.get() with a null guard. AI features now degrade gracefully.
src/modules/ai/copilot.service.ts
Before (crashes on boot)
this.openai = new OpenAI({
  apiKey: config.getOrThrow('OPENAI_API_KEY')
});
After (graceful degradation)
private readonly openai: OpenAI | null;
const key = config.get<string>('OPENAI_API_KEY');
this.openai = key ? new OpenAI({ apiKey: key }) : null;
if (!key) this.logger.warn('AI disabled: no key');
CD Pipeline — .env Overwritten on Every Deploy Action Required
Problem: The CD "Create .env on server" step (cd.yml:127) writes a fresh .env from GitHub secrets on every deploy, overwriting any manual edits (including manually added OPENAI_API_KEY).

Also missing from CD pipeline: PAYPAL_CLIENT_ID, PAYPAL_CLIENT_SECRET, PAYPAL_MODE — these are in the backend but not in the CD's dotenv builder at cd.yml:86-113.

Fix: Add all required keys as GitHub Actions secrets (Repo → Settings → Secrets → Actions → New repository secret).

Rules & Takeaways
Rule 1 — Optional services must never use getOrThrow()
getOrThrow() is for hard infrastructure (DB URL, JWT secrets). External APIs (OpenAI, Stripe, Twilio, SendGrid) are optional — use get() with a null guard and degrade gracefully. A missing OpenAI key should log a warning, not crash the entire app.
Rule 2 — docker restart does NOT reload env_file
Docker caches the environment at container creation time. docker restart crm_api keeps the old env vars. Always use docker compose up -d --no-deps api after changing .env — this recreates the container from the current env_file.
Rule 3 — Read logs FIRST when service works but HTTP fails
When a NestJS service passes unit/integration tests but the HTTP endpoint returns 500, the bug is in a global provider (guard, interceptor, middleware, pipe) that only runs in the HTTP request lifecycle. The global exception filter already logs the full stack trace — check docker logs before any code inspection. Stack trace = file + line + error code = instant root cause.
Rule 4 — CD pipeline is the source of truth for production .env
Never rely on manually edited server files surviving a deploy. Every deploy overwrites .env from GitHub secrets. Add ALL required environment variables as GitHub Actions secrets before deploying. The server file is ephemeral; the secret store is permanent.
Rule 5 — withoutTenantScope() is the correct auth pattern
Auth queries (login, register, token refresh) are cross-tenant by nature — a user's email must be looked up without a tenant filter. All AuthRepository methods correctly use this.prisma.withoutTenantScope(() => ...), which sets tenantId: null in ALS, causing the Prisma extension to bypass scoping entirely. This pattern is verified and correct.
