Issue SSL Certificate

Pending
ℹ

This project uses host-installed certbot with the webroot method. Nginx starts HTTP-only first (no SSL directives in the initial app.conf), certbot completes the ACME challenge, then the CD pipeline renders the full SSL config on first deploy. DNS must be pointed at your server before running these commands.

Step 1 — Install certbot

bash 
apt-get update && apt-get install -y certbot

Step 2 — Start Nginx (HTTP only, no certs needed yet)

bash 
cd /opt/crm-with-blockchain-rag
docker compose up -d nginx
docker exec crm_nginx nginx -t

Step 3 — Get the certbot webroot path

bash 
WEBROOT=$(docker volume inspect crm-with-blockchain-rag_certbot_www \
  --format '{{ .Mountpoint }}')
echo "Webroot: $WEBROOT"

# If the volume name doesn't match, find it with:
docker volume ls | grep certbot

Step 4 — Issue the SSL certificate

bash 
certbot certonly \
  --webroot \
  --webroot-path "$WEBROOT" \
  --non-interactive \
  --agree-tos \
  --email admin@bestpurchasestore.com \
  -d bestpurchasestore.com \
  -d www.bestpurchasestore.com

Step 5 — Verify certs exist

bash 
ls -la /etc/letsencrypt/live/bestpurchasestore.com/
# You should see fullchain.pem and privkey.pem

Step 6 — Set up auto-renew cron

bash 
# Renews on the 1st and 15th of every month at 02:30
# Reloads nginx in-place after renewal (zero-downtime — no restart)
(crontab -l 2>/dev/null; echo "30 2 1,15 * * certbot renew --quiet && docker exec crm_nginx nginx -s reload") | crontab -

# Verify it was added
crontab -l

Step 7 — Trigger the CD pipeline

Push to main or manually trigger the CD workflow in GitHub Actions. The deploy script renders single-slot.conf.template → app.conf with the real cert paths and reloads Nginx. After that, HTTPS is live.

Verify SSL is working

curl -I https://bestpurchasestore.com

Automated Backups & Cron

Pending

Create backup script

bash 
sudo mkdir -p /opt/backups/postgres
sudo nano /opt/backups/backup-db.sh
/opt/backups/backup-db.sh 
#!/bin/bash
set -e
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
BACKUP_DIR=/opt/backups/postgres
mkdir -p $BACKUP_DIR

docker compose -f /opt/nexus-crm/docker-compose.prod.yml exec -T postgres \
  pg_dump -U crm_user crm_db | gzip > "$BACKUP_DIR/crm_db_$TIMESTAMP.sql.gz"

# Keep 30 days
find $BACKUP_DIR -name "*.sql.gz" -mtime +30 -delete
echo "Backup complete: crm_db_$TIMESTAMP.sql.gz"
bash 
sudo chmod +x /opt/backups/backup-db.sh

# Test it
sudo /opt/backups/backup-db.sh

# Set up cron
crontab -e
crontab entries 
# Daily DB backup at 2am
0 2 * * * /opt/backups/backup-db.sh >> /var/log/crm-backup.log 2>&1

# SSL renewal every Monday at 3am
0 3 * * 1 cd /opt/nexus-crm && docker compose -f docker-compose.prod.yml run --rm certbot renew && docker compose -f docker-compose.prod.yml restart nginx >> /var/log/certbot-renew.log 2>&1

⚠️ Known Deployment Gotchas

Issues encountered during first deploy — documented so you don't hit them again.

#1 — Wrong PostgreSQL Docker image (pgvector missing)

Error:
ERROR: extension "vector" is not available DETAIL: Could not open extension control file ".../vector.control": No such file or directory Cause: The postgres:16-alpine image does not include pgvector. The schema uses vector columns for AI embeddings.

Fix: Use pgvector/pgvector:pg16 in docker-compose.yml instead of postgres:16-alpine, then recreate volumes: 
docker compose down -v
docker compose up -d
#2 — CREATE INDEX CONCURRENTLY inside Prisma migration transaction

Error:
ERROR: CREATE INDEX CONCURRENTLY cannot run inside a transaction block (code: 25001) Cause: Prisma wraps migrations in a transaction by default. CREATE INDEX CONCURRENTLY is forbidden inside transactions — it was written with production zero-downtime in mind but is incompatible with Prisma's migration runner.

Fix: Remove CONCURRENTLY from the vector index in prisma/migrations/20260405000001_add_ai_embeddings_and_blockchain/migration.sql: 
-- Before
CREATE INDEX CONCURRENTLY IF NOT EXISTS "ai_embeddings_embedding_idx" ...

-- After
CREATE INDEX IF NOT EXISTS "ai_embeddings_embedding_idx" ...
This is safe on initial deploy since the table is empty — no table lock impact.
#3 — MongoDB missing from dev docker-compose

Error:
MongooseServerSelectionError: connect ECONNREFUSED 127.0.0.1:27017 Cause: The backend's MongooseModule tries to connect on startup and crashes after 9 retries. MongoDB was not in the dev docker-compose.yml.

Fix: MongoDB is now in docker-compose.yml. Always start infra before the app: 
docker compose up -d
# wait for healthy, then start the app
#4 — Health check URL has /api/v1 prefix

The correct health endpoint is /api/v1/health — not /health. The app uses a global API prefix. 
curl http://localhost:3001/api/v1/health
#5 — Wrong backend start script name

Error:
npm error Missing script: "start:prod" Cause: crm-backend/package.json has no start:prod script. The correct production script is npm start → node dist/main. Also triggered by running the command from the wrong directory (crm-frontend/ instead of crm-backend/).

Fix: 
cd /opt/nexus-crm/crm-backend
npm run build
nohup npm start > /tmp/crm-api.log 2>&1 &
#6 — Frontend starts with no .next build directory

Error:
Error: Could not find a production build in the '.next' directory. Cause: Running npm run build && nohup npm start & as a single backgrounded command causes the build to receive SIGTSTP and stop mid-way. npm start then runs against a missing or incomplete .next directory.

Fix: Always run build in the foreground first, then background only the start: 
cd /opt/nexus-crm/crm-frontend
npm run build                                  # foreground — wait for it to finish
nohup npm start > /tmp/crm-web.log 2>&1 &     # background only the server
#7 — Port 3000 already in use on restart (EADDRINUSE)

Error:
Error: listen EADDRINUSE: address already in use :::3000 Cause: A previous npm start process is still running in the background. Kill it before restarting: 
kill $(fuser 3000/tcp 2>/dev/null)
nohup npm start > /tmp/crm-web.log 2>&1 &
#9 — Dashboard crashes with Digest 3978038655 — forwardRef icons passed across RSC boundary

Error:
Functions cannot be passed directly to Client Components... {$$typeof: ..., render: function, displayName: ...} Digest: 3978038655 Cause: DashboardPage (Server Component) passed Lucide icon component references (e.g. icon: Users) as props to StatCard (Client Component). React forwardRef components are functions and cannot be serialized across the server/client boundary.

Fix: Render icons as JSX on the server before passing as props, and accept React.ReactNode instead of LucideIcon: 
// Before (crashes)
icon: Users

// After (works)
icon: <Users size={16} strokeWidth={2} />
Rule: All props crossing Server → Client boundary must be serializable (strings, numbers, plain objects, JSX). Never pass component references or functions.
#8 — Dashboard 400 errors on JS chunks after env change

Error:
Failed to load resource: 400 (Bad Request) _next/static/chunks/…js ChunkLoadError: Loading chunk failed. Cause: NEXT_PUBLIC_* variables are baked into the bundle at build time. After changing .env.local the app must be rebuilt — starting without rebuilding serves mismatched chunks.

Fix: Always rebuild after any .env.local change: 
npm run build
nohup npm start > /tmp/crm-web.log 2>&1 &
#10 — Analytics page 500 on /analytics/revenue — Prisma camelCase column names in raw SQL

Error:
GET /api/v1/analytics/revenue → 500 Internal Server Error Uncaught TypeError: Cannot read properties of undefined (reading 'replace') Cause: Prisma migrations preserve camelCase field names as column names exactly — wonAt, tenantId, ownerId, firstName, etc. Raw $queryRaw bypasses the ORM, so snake_case names like won_at, tenant_id, owner_id fail silently in PostgreSQL (column not found → 500). The frontend TypeError was a downstream effect: undefined data passed to chart formatters.

Three additional shape mismatches were also fixed at the same time:
  • getLeadSourceBreakdown returned raw Prisma { _count: { id } } — frontend expected { count, percentage }
  • getPipelineFunnel returned { stageName, dealCount } + used a route param (:pipelineId) — frontend sent a query param and expected { stage, count, value, conversionRate }
  • getSalesPerformance returned { user: { firstName }, deals } — frontend expected { userId, name, dealsWon, leadsConverted }
Fix — raw SQL column names (analytics.service.ts): 
// Before (crashes — snake_case does not exist in DB)
TO_CHAR("won_at", 'YYYY-MM') AS month_key
WHERE tenant_id = ${tenantId}
  AND won_at   >= ${startDate}
JOIN users u ON u.id = d.owner_id

// After (correct — Prisma keeps camelCase column names)
TO_CHAR("wonAt", 'YYYY-MM') AS month_key
WHERE "tenantId" = ${tenantId}
  AND "wonAt"   >= ${startDate}
JOIN users u ON u.id = d."ownerId"
Fix — controller route (analytics.controller.ts): 
// Before
@Get('pipeline-funnel/:pipelineId')
getPipelineFunnel(@Param('pipelineId') id: string)

// After
@Get('pipeline-funnel')
getPipelineFunnel(@Query('pipelineId') id: string)
Rule: Before writing any $queryRaw, grep the migration SQL to confirm exact column names. Prisma never auto-converts camelCase to snake_case — only @map("snake_case") does that. Always verify the shape your service returns matches the TypeScript interface the frontend imports.
#11 — Leads / Deals / Tasks / Tickets return 400 on every page load — empty string query params fail Zod enum validation

Error:
GET /api/v1/leads?page=1&limit=20&search=&status=&source= → 400 Validation failed Cause: The frontend sends every filter param on each request, including unset ones as "" (empty string). Zod's z.enum([...]).optional() accepts undefined but rejects "" — empty string is not a valid enum member.

Fix — wrap enum/uuid filter fields with z.preprocess in the DTO schemas: 
// Before — crashes on status=&source=
status: z.enum(['NEW', 'CONTACTED', ...]).optional()

// After — empty string coerced to undefined before validation
status: z.preprocess(
  (v) => (v === '' ? undefined : v),
  z.enum(['NEW', 'CONTACTED', ...]).optional()
)
Files changed: leads.dto.ts, deals.dto.ts, tasks.dto.ts, tickets.dto.ts — all FilterXxxSchema enum and uuid fields.

Rule: HTTP query params are always strings. .optional() only accepts undefined, not "". Always preprocess enum/uuid filter fields to convert empty strings to undefined.
