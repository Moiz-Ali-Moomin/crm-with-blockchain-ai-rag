BUG-04
ActivityFeed dynamic color classes may be purged by Tailwind in production
src/app/(dashboard)/dashboard/page.tsx · lines 28–33
● High

Description

The MOCK_ACTIVITY array stores color strings like 'bg-emerald-500', 'bg-blue-500', 'bg-violet-500', 'bg-amber-500' as data. These are interpolated into className at runtime. Tailwind's JIT scanner cannot detect them at build time and will purge them from the production CSS bundle.

Impact

In production, all activity avatars render with no background color (transparent/no class). Works in development only. The bug is invisible during development and CI.

dashboard/page.tsx — lines 28–32
const MOCK_ACTIVITY = [
  { ..., color: 'bg-emerald-500' },   // ← dynamic class, will be purged
  { ..., color: 'bg-blue-500' },
  { ..., color: 'bg-violet-500' },
  { ..., color: 'bg-amber-500' },
] as const;

Fix

Either add all dynamic classes to the safelist in tailwind.config.ts, or (preferred) map to full class strings inside the component where Tailwind can statically detect them:

// tailwind.config.ts
safelist: ['bg-emerald-500', 'bg-blue-500', 'bg-violet-500', 'bg-amber-500']

// Or replace color string with a semantic key and map it:
const colorMap = {
  green:  'bg-emerald-500',
  blue:   'bg-blue-500',
  purple: 'bg-violet-500',
  yellow: 'bg-amber-500',
} as const;

INC-02
CD pipeline fails with "container name already in use" on every re-deploy
Date: 2026-04-14 · File: .github/workflows/cd.yml · Type: Pipeline bug / idempotency
⚠ Pipeline Bug

Error

GitHub Actions — deploy step output
Error response from daemon: Conflict. The container name "/crm_web_green"
is already in use by container "f906c0a17134...".
You have to remove (or rename) that container to be able to reuse that name.

Root cause

The deploy script determined the idle slot (e.g. green) and immediately ran docker compose up against it. It assumed the idle slot was always empty. On a second deploy — or after any failed deploy that left containers behind — the containers crm_api_green and crm_web_green already existed. Docker cannot create a new container with the same name as an existing one, so the pipeline errored out before any new image was started.

Impact

Every CD run after the first deploy was broken. The pipeline could not recover without manual docker rm on the server. Zero-downtime deploys were effectively non-functional beyond the initial deploy.

Fix applied — .github/workflows/cd.yml

Inserted an idempotent cleanup block before the docker compose up for the idle slot (between steps 5a and 5c, now labelled 5b). Two layers of safety:

.github/workflows/cd.yml — inserted at step [5b]
# Layer 1: compose down — handles containers tracked by compose project
docker compose \
  -f "docker-compose.${IDLE}.yml" \
  --project-name "crm-${IDLE}" \
  down --remove-orphans || true

# Layer 2: force-remove by name — handles containers created outside compose
# (e.g. manually via docker run, or from a different compose project context)
docker rm -f "crm_api_${IDLE}" "crm_web_${IDLE}" 2>/dev/null || true

Why both layers are needed

1

compose down only removes containers it knows about — tracked under the crm-${IDLE} project label. If a container was created manually or under a different project name, compose ignores it.

2

docker rm -f by name catches everything compose misses — any container with that exact name regardless of how it was created. The 2>/dev/null || true makes it a no-op when the containers don't exist (first deploy).

Result

Deploy is now fully idempotent. Re-running the pipeline, retrying a failed deploy, or deploying after a manual hotfix all succeed without any manual server intervention.

INC-03
Deployment state drift — .active_slot file lies about which slot is live
Date: 2026-04-14 · File: .github/workflows/cd.yml · Type: State management / idempotency
⚠ Pipeline Bug

Problem

cd.yml — broken slot detection (old)
# Reads a file that can be stale, missing, or just wrong
ACTIVE=$(cat "$DEPLOY_PATH"/.active_slot 2>/dev/null || echo "blue")
if [ "$ACTIVE" = "blue" ]; then
  IDLE="green"
else
  IDLE="blue"
fi

Root cause

The CD script derived the active/idle slot by reading a .active_slot file written at the end of each successful deploy. Any failure scenario — a deploy that crashed after containers started but before the file was written, a manual hotfix, or a server restart — left the file out of sync with actual running containers. On the next deploy the script would target the wrong slot, potentially deploying to the live-serving slot or trying to keep the wrong slot as active.

Impact

Wrong slot selected on deploy. In the worst case the script tears down the actively-serving slot and deploys to it while the supposedly-idle slot (now serving traffic) is untouched. Zero-downtime guarantee broken. No visible error — the script proceeds normally with incorrect state.

Fix applied — runtime detection from Docker process state

cd.yml — runtime slot detection (new)
# Derive truth from what is ACTUALLY running — no file dependency
if docker ps --format '{{.Names}}' | grep -q '^crm_api_green$'; then
  ACTIVE="green"
else
  # Blue is the safe default — first deploy or after a full restart
  ACTIVE="blue"
fi
if [ "$ACTIVE" = "blue" ]; then IDLE="green"; else IDLE="blue"; fi
echo "  Runtime-detected: ACTIVE=$ACTIVE  IDLE=$IDLE"

Result

Slot detection is now stateless and self-healing. It cannot drift from reality because it reads live Docker process state on every deploy. The .active_slot file write at the end of the deploy script was also removed — it served no further purpose.

INC-04
Weak CI→CD contract — CD recomputes image tag instead of consuming CI artifact
Date: 2026-04-14 · File: .github/workflows/cd.yml · Type: Pipeline contract / correctness
⚠ Pipeline Bug

Problem

cd.yml — tag recomputed in CD (old)
elif [ "${{ github.event_name }}" = "workflow_run" ]; then
  SHA="${{ github.event.workflow_run.head_sha }}"
  TAG="sha-${SHA:0:7}"    # ← recomputed, not consumed from CI output
fi

Root cause

The CD workflow recomputed the image tag by taking head_sha from the workflow_run event and slicing it to 7 chars. This matched the CI tagging logic most of the time, but CI used git rev-parse --short HEAD which can produce 8+ chars for disambiguation on large repos. Any mismatch means CD tries to pull a tag that doesn't exist in GHCR — causing the pull step to fail or, worse, falling back to a cached local image from a previous run.

Impact

Risk of deploying the wrong image. The tag that CI built and validated is not guaranteed to be the same string CD attempts to pull. On repositories where git short SHA length auto-extends beyond 7 chars, this breaks silently or causes a hard pull failure.

Fix applied — explicit CI→CD contract with source annotation

cd.yml — hardened tag resolution (new)
if [ -n "${{ github.event.inputs.image_tag }}" ]; then
  # Manual dispatch — explicit override for hotfixes/rollbacks
  TAG="${{ github.event.inputs.image_tag }}"
  echo "  Source: manual override"
elif [ "${{ github.event_name }}" = "workflow_run" ]; then
  # Strict CI contract: use the SHA that CI actually built
  SHA="${{ github.event.workflow_run.head_sha }}"
  TAG="sha-${SHA:0:7}"
  echo "  Source: CI workflow_run head_sha=${SHA}"
else
  echo "ERROR: Cannot determine image tag" && exit 1
fi

Result

The tag source is now explicit and auditable in the deploy log. The head_sha from the triggering CI run is the authoritative source — the same SHA CI used when it built and pushed the image. Manual override is only for emergency use.

INC-05
CI builds Docker images but never validates they actually start — runtime crashes reach CD
Date: 2026-04-14 · File: .github/workflows/ci.yml · Type: Missing quality gate
⚠ Gap

Root cause

The build-and-push job built and pushed the Docker images to GHCR, then exited. A successful docker build only proves the image was assembled — it does not prove the process inside the container can start, read its configuration, bind its port, or respond to requests. A startup crash (bad env var, missing file, broken import) would only be discovered on the production server after the container was already deployed.

Impact

Broken images can reach the production deploy step. The CD health-check would eventually catch it, but only after a new slot was started and the old slot potentially torn down — requiring a manual rollback. CI should be the last line of defence, not CD.

Fix applied — runtime smoke test after push

ci.yml — smoke test step (new, appended to build-and-push)
# Start the API container with minimal env (no real DB needed for /health/live)
docker run -d \
  --name smoke_api \
  --health-cmd "wget -qO- http://127.0.0.1:3001/api/v1/health/live || exit 1" \
  --health-interval 5s --health-timeout 5s \
  --health-retries 12 --health-start-period 20s \
  -e NODE_ENV=production -e PORT=3001 \
  -e DATABASE_URL="postgresql://dummy:dummy@localhost:5432/dummy" \
  -e REDIS_URL="redis://localhost:6379" \
  -e JWT_SECRET="ci-smoke-test-secret-not-for-prod" \
  -p 3001:3001 \
  "${IMAGE_API}:${IMAGE_TAG}"

# Poll Docker health status — fail CI if unhealthy or timeout
for i in $(seq 1 18); do
  STATUS=$(docker inspect --format='{{.State.Health.Status}}' smoke_api)
  [ "$STATUS" = "healthy" ] && docker rm -f smoke_api && exit 0
  [ "$STATUS" = "unhealthy" ] && docker logs smoke_api --tail 50 && docker rm -f smoke_api && exit 1
  sleep 5
done
docker logs smoke_api --tail 50 && docker rm -f smoke_api && exit 1

Result

Every image pushed to GHCR is now validated to actually start and serve /api/v1/health/live before it can be consumed by CD. Startup crashes, missing env vars, and broken health endpoints are caught in CI. Container logs are dumped on failure for fast diagnosis.

INC-06
Nginx refuses to start when Grafana container is absent — "host not found in upstream"
Date: 2026-04-14 · Files: nginx/conf.d/single-slot.conf.template, canary.conf.template · Type: Hard dependency / startup failure
⚠ Incident

Error

nginx startup log
nginx: [emerg] host not found in upstream "crm_grafana" in /etc/nginx/conf.d/app.conf:52
nginx: configuration file /etc/nginx/nginx.conf test failed

Root cause

Both single-slot.conf.template and canary.conf.template contained a proxy_pass http://crm_grafana:3000/ directive for the /grafana/ location. Nginx resolves all upstream hostnames at startup time (not request time). If crm_grafana is not running or not on the network at boot, nginx fails its config test and refuses to start entirely — taking down all traffic routing, not just the Grafana path.

Impact

Nginx could not reload or restart whenever the observability stack was down, even partially. This happened in INC-01 (Grafana deliberately excluded from the manual recovery) and would recur on any environment where the observability stack is optional or starts after nginx.

Fix applied — variable upstream with Docker embedded DNS

single-slot.conf.template & canary.conf.template — Grafana block (new)
# Docker embedded DNS — resolves at request time, not startup
resolver 127.0.0.11 valid=10s ipv6=off;
set $grafana_upstream http://crm_grafana:3000;

location /grafana/ {
    proxy_pass $grafana_upstream/;
    proxy_connect_timeout 5s;
    proxy_read_timeout    30s;
    proxy_intercept_errors on;
    # Clean 503 instead of cryptic upstream error
    error_page 502 503 504 = @grafana_unavailable;
}

location @grafana_unavailable {
    add_header Content-Type text/plain always;
    return 503 "Grafana is not available in this environment.\n";
}

Why set $var works

1

When the upstream is a literal string (proxy_pass http://crm_grafana:3000), nginx resolves the hostname once at startup using getaddrinfo(). If DNS fails, the whole config fails.

2

When the upstream is a variable (set $var ...; proxy_pass $var), nginx defers resolution to request time via the configured resolver. 127.0.0.11 is Docker's embedded DNS server — it resolves container names dynamically. A missing container returns NXDOMAIN at request time, triggering the error_page handler instead of aborting startup.

Result

Nginx starts and reloads cleanly regardless of whether crm_grafana is running. Users hitting /grafana/ when Grafana is down receive a clean HTTP 503 with a human-readable message. All other routes are unaffected.

INC-07

Date: 2026-04-14 · File: .github/workflows/cd.yml · Type: Missing pre-flight validation
⚠ Pipeline Bug

Root cause

The CD script checked only whether the crm_internal Docker network existed before deciding to skip the infra bootstrap. INC-09
Smoke test crashes at startup — localhost:5432 unreachable inside Docker container
Date: 2026-04-14 · File: .github/workflows/ci.yml (build-and-push / smoke test step) · Type: CI networking / DB connectivity
⚠ Pipeline Bug

Error

API container stdout — startup crash
Error: P1001: Can't reach database server at `localhost:5432`
Please make sure your database server is running at `localhost:5432`.

Root cause — three compounding issues

1. Wrong hostname. The smoke test passed DATABASE_URL=postgresql://dummy:dummy@localhost:5432/dummy. Inside a Docker container, localhost resolves to the container's own loopback interface — not the runner host, not any sibling container. There is no Postgres process inside the API container, so the connect attempt fails immediately.

2. Bootstrap-time connection. PrismaService.onModuleInit() calls await this.$connect(). NestJS runs all OnModuleInit hooks during application bootstrap, before binding the HTTP port. The process crashes before port 3001 is even open, so /health/live is never reachable regardless of wait time.

3. Wrong Postgres image. The schema declares extensions = [pgvector(map: "vector")]. prisma migrate deploy executes CREATE EXTENSION vector — a statement that requires the pgvector shared library. Plain postgres:16 does not ship it. Migrations fail with "could not open extension control file" if the wrong image is used.

Impact

Every push to main fails at the smoke test step. The image is pushed to GHCR (build succeeds) but CI exits non-zero, blocking CD from triggering. The pipeline was effectively dead on main after the smoke test was introduced.

Fix applied — real Postgres on a shared Docker network

ci.yml — smoke test (new, key steps)
# [1] Isolated bridge network — containers resolve each other by --name
docker network create smoke_net

# [2] pgvector image — required for CREATE EXTENSION vector in migrations
docker run -d --name smoke_postgres --network smoke_net \
  -e POSTGRES_USER=test -e POSTGRES_PASSWORD=test -e POSTGRES_DB=test \
  pgvector/pgvector:pg16

# [3] Wait for Postgres (pg_isready inside the container — no host psql needed)
docker exec smoke_postgres pg_isready -U test -d test -q

# [4] Run migrations from the built image — same artifact as deploy
docker run --rm --network smoke_net \
  -e DATABASE_URL="postgresql://test:test@smoke_postgres:5432/test" \
  --entrypoint npx "${IMAGE_API}:${IMAGE_TAG}" prisma migrate deploy

# [5] Start API — hostname "smoke_postgres" resolves inside smoke_net
docker run -d --name smoke_api --network smoke_net \
  -e DATABASE_URL="postgresql://test:test@smoke_postgres:5432/test" \
  -e JWT_SECRET="ci-smoke-test-not-for-prod-pad-to-32-chars" \
  -e JWT_REFRESH_SECRET="ci-smoke-refresh-not-for-prod-pad-32c" \
  -p 3001:3001 "${IMAGE_API}:${IMAGE_TAG}"

# [6] curl from host via mapped port — validates full network stack
curl -sf http://localhost:3001/api/v1/health/live

Why each decision was made

1

Named Docker network, not GitHub Actions services: — GitHub services are reachable from the runner host at localhost:<port>, but NOT from inside another Docker container. A shared bridge network lets containers find each other by --name, which is how production networking works.

2

pgvector/pgvector:pg16 not postgres:16 — schema.prisma declares extensions = [pgvector]. prisma migrate deploy runs CREATE EXTENSION vector. The pgvector shared library (vector.so) is not present in the official Postgres image. Using the wrong image causes a migration failure, not a connect failure — a different and harder-to-diagnose error.

3

Migrations run from the built image — docker run --rm --entrypoint npx <image> prisma migrate deploy uses the exact migration files baked into the CI artifact. This catches schema drift and broken migrations before the image reaches the server.

4

curl from host via -p 3001:3001 — exercises the container's port binding and the kernel's TCP listen backlog. More realistic than docker exec wget 127.0.0.1 which bypasses all of that and can return 200 even when external connectivity is broken.

5

Both /health/live and /health/ready probed — /live is the gate (process up, port bound). /ready confirms Prisma can actually query the test DB after migrations. A warning (not a failure) is logged if /ready is non-200 so it doesn't mask unrelated issues.

Correct DATABASE_URL for smoke test context

# WRONG — "localhost" inside a container = container loopback, not host

Result

The smoke test now starts a real Postgres, applies migrations, boots the API, and validates both /health/live (process up) and /health/ready (DB connected). INC-10
Smoke test crash — getOrThrow("MONGO_URI") throws at bootstrap, port never opens, health check returns HTTP 000
Date: 2026-04-14 · Files: src/core/database/mongo.module.ts · src/modules/ai/rag.service.ts · .github/workflows/ci.yml · Type: Eager dependency / bootstrap crash
⚠ Pipeline Bug

Error

API container stdout — startup crash
Configuration key "MONGO_URI" does not exist
TypeError: Configuration key "MONGO_URI" does not exist
    at ConfigService.getOrThrow (...)
    at MongooseModule.forRootAsync.useFactory (mongo.module.ts:32)
    at NestFactory.create (...)

# Health check returns 000 because curl cannot connect — port 3001 was
# never bound. The process exited during bootstrap, before listen().

Root cause — two independent bootstrap crashes

1. mongo.module.ts:32 — config.getOrThrow('MONGO_URI')
MongooseModule.forRootAsync factory runs synchronously during NestJS DI graph construction. getOrThrow throws immediately when the key is absent. NestJS catches the exception during module initialisation, logs it, and exits the process — before app.listen() is ever called. Port 3001 is never opened. curl gets connection refused → HTTP 000, not 500.

2. rag.service.ts:91 — config.getOrThrow('OPENAI_API_KEY')
Same pattern in the RagService constructor. Even if MONGO_URI were fixed, the absence of OPENAI_API_KEY would produce an identical crash on the next bootstrap attempt. Both keys are declared z.optional() in env.validation.ts — Zod allows them to be absent, but the services immediately override that contract by using getOrThrow.

Why HTTP 000, not 500

HTTP 000 is curl's sentinel for no response received — the TCP connection itself was refused (ECONNREFUSED) or timed out. This only happens when no process is listening on the target port. It is categorically different from an HTTP 500 (server running, handler threw). The process died during NestJS bootstrap before app.listen(3001), so the port is closed. curl never exchanges a single byte — it exits with code 7 (failed to connect) and writes "000" via -w "%{http_code}".

Bootstrap sequence — where the crash occurs

1

NestJS resolves the DI graph. AppModule → CoreModule → MongoModule.

2

MongooseModule.forRootAsync factory executes. config.getOrThrow('MONGO_URI') throws synchronously — MONGO_URI is not in the environment.

3

NestJS propagates the exception, logs "Configuration key MONGO_URI does not exist", and terminates the process via process.exit(1).

4

app.listen(3001) is never reached. Port 3001 is never opened. The Docker healthcheck and the CI curl both get ECONNREFUSED → HTTP 000.

Fix 1 — Production (recommended) · mongo.module.ts

Replace getOrThrow with get and degrade gracefully when MONGO_URI is absent. MongoDB backs AI audit logs — fire-and-forget writes. The core CRM (Postgres) must not be held hostage by an optional secondary datastore.

src/core/database/mongo.module.ts — useFactory (fixed)
useFactory: (config: ConfigService) => {
  const uri = config.get<string>('MONGO_URI');   // get(), not getOrThrow()

  if (!uri) {
    logger.warn(
      'MONGO_URI is not set — MongoDB disabled. ' +
      'AI audit logging and RAG features will not function.'
    );
    // Sentinel URI: Mongoose attempts connection, fails fast (1s timeout),
    // logs an error — but does NOT throw, does NOT crash the process.
    return {
      uri: 'mongodb://127.0.0.1:27017/__disabled__',
      serverSelectionTimeoutMS: 1000,
      connectTimeoutMS: 1000,
    };
  }

  return { uri, maxPoolSize: 10, minPoolSize: 2, ... };
}

Architecture improvements — what this incident reveals

1

env.validation.ts and service constructors disagree. Zod marks MONGO_URI and OPENAI_API_KEY as z.optional(), but service code uses getOrThrow. The contract should be consistent: if it's optional in the schema, every consumer must handle its absence. Audit all getOrThrow calls — only use it for keys that are z.string() (required) in the schema.

2

Feature modules should be conditionally loaded. AiModule and MongoModule are always imported regardless of whether AI features are enabled. Use a feature-flag pattern: ...(process.env.MONGO_URI ? [MongoModule, AiModule] : []) in CoreModule and AppModule. Cleaner than sentinel URIs and avoids Mongoose error noise entirely.

3

/health/live must be dependency-free — enforce it structurally. The liveness endpoint currently works correctly (no DB calls), but it depends on the process surviving bootstrap. Any constructor-level getOrThrow for an optional var can silence the liveness probe forever. Consider a minimal LivenessModule that imports nothing from CoreModule and is registered first.

4

Smoke test should enumerate and explain every env var. An undocumented docker run with 4 env vars that silently omits required ones creates exactly this class of bug. The updated CI snippet (above) comments each var against its Zod schema type so future maintainers know why each is or isn't included.
